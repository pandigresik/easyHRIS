Push Protocol

Intro

Perangkat ZKTeco yang support ADMS/WDMS/Web Service menggunakan protocol yang namanya Push Protocol. Berikut cara kerjanya.

Mesin

Mesin yang dapat dikoneksikan harus mempunyai fitur koneksi jaringan baik melalui ethernet ataupun wireless. Mesin pun harus punya fitur kirim data ke web atau cloud. Cara koneksi dengan masuk menu COMM setting.

Fungsi Utama

Ada 6 fungsi utama push protocol (sudut pandang dari sisi client):

  1. Initializing Information Exchange (Inisialisasi Pertukaran Informasi antara client dan server)
  2. Uploading Update Information (Mengirim Informasi Terkini)
  3. Uploading Data (Mengirim Data)
  4. Downloading Command (Mengambil Perintah dari Pusat)
  5. Command Reply (Membalas Perintah dari Pusat)
  6. Remote Attendance (Laporan Kehadiran Jarak Jauh/tidak di tempat)

Proses Komunikasi

Fungsi-fungsi no 2-6 di atas hanya bisa lanjut kalau proses nomor 1 sukses dulu.

Contoh proses:

  1. Handshake. Client dapat info konfig dari server.

    CLIENT        ------------>         SERVER
GET /iclock/cdata?SN=OID7054297041500253&options=all&pushver=2.2.14&language=73

CLIENT        <------------         SERVER
GET OPTION FROM: 354313
Stamp=82983982
OpStamp=9238883

  2. Device ambil perintah yang diminta dari server. Dalam hal ini mengubah pin atas nama Wawan

    CLIENT        ------------>         SERVER
GET /iclock/getrequest?SN=OID7054297041500253

CLIENT        <------------         SERVER
C:122:INFO
C:123:DATA USER PIN=852 Name=Wawan

  3. Device mengirimkan jawaban ke server setelah melaksanakan komando

    CLIENT        ------------>         SERVER
POST /iclock/devicemd?SN=OID7054297041500253

CLIENT        <------------         SERVER
OK

  4. Device mengirimkan data ke server jika misal ada data tambahan atau ada data kehadiran baru masuk

    CLIENT        ------------>         SERVER
POST /iclock/cdata?SN=OID7054297041500253&Stamp=26

CLIENT        <------------         SERVER
OK xx

    Keterangan: Server memberi info balik ke client. xx menunjukkan berapa jumlah baris data yg diterima server pada command terakhir.

Penjelasan Fungsi-Fungsi Utama

Inisialisasi Pertukaran Informasi antara Client dan Server

Contoh data dari client:

GET /iclock/cdata?SN=0316144680030&options=all&pushver=2.2.14&language=83&pushcommkey=429594af1641269779559e8554b5d126 HTTP/1.1
Host: 58.250.50.81:8011
User-Agent: Clock Proxy/1.09
Connection: close
Accept: */*

Contoh jawaban dari server:

HITP/1.1 200 OK
Server:nginx/1.6.0
Date: Fri, 03 Jul 2015 06:53:01 GMT
Content-Type: text/plain
Content-Length: 190
Connection: close
Pragmar no-cache
Cache-Control: no-store

GET OPTION FROM: 0316144680030
ATTLOGStamp=None
OPERLOGStamp=9999
ATTPHOTOStamp=None

TransTimes=00:00;14: 05
Transinterval=1
TransFlag=TransData AttLog OpLog AttPhoto EnrollUser ChgUser EnrollFP ChgFP UserPic

Keterangan:

  • Date header digunakan untuk sinkronisasi waktu client

  • Baris GET OPTION FROM: {{ serialNumber }}, diikuti konfigurasi dipisahkan \n\r

  • Baris {{ logType }}Stamp alternatif konfignya: ATTLOG Attendance record OPERLOG Operation log ATTPHOTO Attendance photo Ketika client upload suatu data, tanda timestampnya diupload sebagai fungsi cut-off. Server harus simpan ini. Saat alat reboot, client meminta inisialisasi pertukaran data kembali. Nah server harus mengembalikan stamp terakir dari client agar pengiriman data berikutnya hanya yang baru yg dikirim (nggak semua diupload dari awal).

  • Konfigurasi ErrorDelay: berapa detik interval mengirim ulang data jika terjadi error.

  • Konfigurasi TransTimes: Waktu kapan client mengecek dan mengirimkan kembali data secara periodik, dalam format 24 jam (hour:minute).

  • Konfigurasi TransInterval: Interval untuk client mengecek dan mengirim data baru (dalam menit). Jika 0, tidak ada data dikirim

  • Konfigurasi TransFlag: Identifikasi data yang diminta dari client. Ada dua format,

    1. Format pertama yang pertama pakai bit, contoh:

      TransFlag = 1111000000

      Yang artinya:

      Digit keData yang dimintaNilai
      1Attendance recordYa
      2Operation logYa
      3Attendance photoYa
      4Enrolling a new userYa
      5Changing user informationTidak
      6Enrolling a new fingerprintTidak
      7Changing a fingerprintTidak
      8Fingesprint imageTidak
      9New enrolled faceTidak
      10User pictureTidak

    2. Format kedua yang pertama pakai string, contoh:

      TransFlag = TransData AttLog<tab>OpLog<tab>AtPhoto...<dst>..

      Di mana data yang diminta dicantumkan dengan string dipisahkan dengan tab:

      StringDeskripsi Data Diminta
      AttLogAttendance log
      OpLogOperation log
      AttPhotoAttendance photo
      EnrollUserEnrolling a new user
      ChgUserChanging user information
      EnrollfPEnrolling a new fingerprint
      ChgFPChanging a fingerprint
      FPImagFingerprint image
      FACENew enrolled face
      UserPicUserpicture

  • Konfigurasi Timezone: sesuaikan agar client record jamnya tepat. Gunakan 7 untuk WIB, 8 WITA, 9 WIT.

  • Konfigurasi ServerVer: set 2.2.14.

  • Konfigurasi Realtime: set 1 jika ingin mendapatkan realtime update dari mesin.

Khusus untuk nomor 10, formatnya berupa 3 angka fungsi, tiap digitnya 0—tidak mendukung fungsi, 1—mendukung fungsi. Jadi jika misalnya 101 artinya:

Digit keData yang dimintaNilai
1Fingerprint functionYa
2Face functionTidak
3User photo functionYa

Mengirim Data

Pengiriman data bisa 3 cara:

  • Realtime uploading (langsung) Untuk realtime, beritahu saat inisialisasi.
  • Interval uploading (berjeda) Untuk interval, yang digunakan adalah parameter TransInterval.
  • Timed uploading (terjadwal) Untuk pengiriman terjadwal, yang digunakan adalah parameter TransTimes.

Keterangan lebih lanjut baca kembali terminologi di atas dan cara settingnya di bab Inisialisasi.

Mengambil Perintah dari Pusat

Jika server ingin melakukan operasional dari pusat, server membuat format command, tunggu sampai mesin memulai koneksi, lalu server membalas dengan perintah ke mesin

Contoh request dari client:

Get /iclock/getrequest?SN=0316144680030
Host: 58.250.50.81:8011
..dst..

Contoh jawaban dari server, jika tidak ada perintah dikirimkan, maka jawabannya sbb:

HTTP/1.1 200 OK
Date: xxx
Content-Length: 2

OK

Jika ada perintah dikirimkan, jawabannnya sbb:

HTTP/1.1 200 OK 
Date: xxx
Content-Length: xxx

<data command>

Tag data command kelak diisi sesuai peruntukannya.

Command DATA

Command UPDATE

  1. Perintah Update Data User Format perintahnya untuk tiap baris:

    C:{{CmdId}}:DATA<spasi>UPDATE<spasi>USERINFO<spasi>PIN={{UserID}}<tab>Name={{Nama}}<tab>Passwd={{passwd}}<tab>Card={{card}}<tab>Grp={{group}}<tab>TZ={{timePeriod}}<tab>Pri={{userPrivillegeValue}}

    Keterangan:

    • PIN menunjukkan UserId
    • Group menunjukkan termasuk group apa ybs
    • TZ menunjukan time period yang digunakan oleh user.
      • Digit 1-4 --> group time period
      • Digit 5-8 --> Personal time period 1
      • Digit 9-12 --> Personal time period 2
      • Digit 13-16 --> Personal time period 3
    • Pri menunjukkan privillege user:
      • 0 Normal User
      • 2 Registrar
      • 6 Admin
      • 10 User-defined
      • 14 Super Admin

  2. Perintah Update Fingerprint Data TBD

  3. Perintah Update Face Template TBD

  4. Perintah Update User Photo

    C:{{CmdId}}:DATA<spasi>UPDATE<spasi>USERPIC<spasi>PIN={{UserId}}<tab>Size={{Size}}<tab>Content={{fileContent}}

    Keterangan:

    • Size: Panjang file dalam format base64
    • Content: Isi file setelah diencode dalam base64

  5. Perintah Kirim Pesan Singkat

    C:{{CmdId}}:DATA<spasi>UPDATE<spasi>SMS<spasi>MSG={{Message}}<tab>TAG={{Tag}}<tab>UID={{MessageNumber}}<tab>MIN={{Duration}}<tab>StartTime={{StartTime}}

    Keterangan:

    • TAG diisi nilai sesuai jenisnya:
      • 253 Public short message
      • 254 User short message
      • 255 Reserved short message
    • UID diisi nomor dari SMS (integer)
    • MIN diisi menit durasi ditampilkannya short message
    • Start time diiisi kapan mulai ditampilkannya pesan dalam format YYYY-MM-DD HH:MM:SS

  6. Perintah Kirim Pesan-Pesan Singkat Personal, untuk tiap baris:

    C:{{CmdId}}:DATA<spasi>UPDATE<spasi>USER_SMS<spasi>PIN={{UserId}}<tab>UID={{MessageNumber}}

Enrollment (Pendaftaran Fingerprint) secara Remote

Pendaftaran fingerprint diinisiasi dari server, dan dilakukan di client. Perintahnya dari server:

C:{{CmdId}}:ENROLL_FP<spasi>PIN={{UserId}}<tab>FID={{FingerPrintID}}<tab>RETRY={{NumberOfRetry}}<tab>OVERWRITE={{OverwriteExisting}}

Keterangan:

  • PIN menunjukkan UserId
  • FID menunjukkan nomor fingerprint
  • RETRY menunjukkan berapa kali harus coba ulang kalau enroll gagal
  • OVERWRITE menunjukkan apakah fingerprint yang ada harus ditimpa atau tidak. 0 --> tidak usah 1 --> timpa

Jawaban mesin:

ID={{CmdId}}&Return={{ErrorCode}}&CMD=ENROLL_FP
